# Design: Flash Idea - 零摩擦语音捕捉管道

## Problem Statement

需要在 Android 手机（OPPO Find X7 Ultra）上实现零摩擦的想法捕捉。现有方案全部不够快：
- 飞书原生 App：启动慢、层级深、开屏加载
- 飞书录音豆：同步慢，提取原文要点好多层
- 手机语音助手（小布）：语义理解层引入不必要的延迟和失败
- 微信文件传输助手 / 飞书个人会话：无法添加桌面快捷方式直达

核心需求：点击图标 → 聊天气泡界面 → 语音转文字（豆包输入法）→ 发送 → 自动追加到飞书云文档。冷启动 < 500ms。

## What Makes This Cool

不是"又一个笔记 App"，而是一个 **personal idea ingress pipe（想法入口管道）**。

每条消息是一个结构化事件，飞书云文档只是第一个"接收端"（sink）。管道设计好之后，未来可以接 AI 分类、日报汇总、OpenClaw、Obsidian 等任何下游工具。

用户原话：
> "我觉得我们应该把这个管道做得很好，就是这个管道方便后续的东西来接，而不是我们一个管道企图取代整个后续处理的流程"

## Constraints

- 只给自己用，不需要多用户、登录、权限管理
- 冷启动 < 500ms（必须实测验证，不是假设）
- 语音识别依赖系统输入法（豆包），不内置 ASR
- 飞书基础免费版 10,000 次 API 调用/月，个人使用足够
- 优先 Android，macOS 为次要平台
- 开源友好的架构设计

## Premises

1. 必须是原生 App 形态，PWA 或快捷指令不够快
2. Tauri 2.0 是正确的跨平台选择（Rust + Web，轻量，覆盖 Android + macOS）
3. 飞书云文档是正确的存储终点
4. 语音输入交给系统输入法（豆包），不自己做识别

## Cross-Model Perspective

Codex 独立 cold read 提出的关键洞察：

- **最酷的版本**：每次提交不是存一段文本，而是创建一个"持久化想法数据包"（durable idea packet），带有结构化的 context（设备、时区、输入方式、同步目标、状态）。飞书只是第一个 sink。
- **核心兴奋点**：不是"更快的笔记 App"，而是"拥有从捕捉到下游的管道"
- **50% 工具**：Feishu CLI / @larksuiteoapi/lark-mcp 作为参考实现理解 API 模式，但不作为运行时依赖
- **Weekend build 建议**：Day 1 搭 Tauri Android 单屏 + SQLite 本地写入；Day 2 接飞书同步 + 冷启动性能 logging

## Approaches Considered

### Approach A: Tauri 2.0 先跑通 + 冷启动测量 (CHOSEN)

- **Summary**: 用一个周末搭建完整 happy path（输入 → SQLite → 飞书追加），重点测量冷启动各阶段耗时
- **Effort**: S（1-2 天）
- **Risk**: Medium（Tauri WebView 冷启动可能超标）
- **Pros**: 最快验证核心假设；成功则零浪费继续开发；Codex 建议的冷启动 logging 能给出精确数据
- **Cons**: 如果冷启动超标，需要切换到方案 B
- **Reuses**: Tauri 2.0 官方 Android 模板，飞书 API 文档中的追加内容示例

### Approach B: 安卓原生 Kotlin + macOS Tauri 双线

- **Summary**: Android 用 Kotlin/Jetpack Compose 保证极致冷启动，macOS 用 Tauri
- **Effort**: M（3-5 天初版）
- **Risk**: Low（技术成熟度高）
- **Pros**: 冷启动确定最快（~200ms）；原生能力全部可用
- **Cons**: 两套代码维护成本高；Kotlin + Rust + Web 三语言混合

### Approach C: Web + Rust CLI 分离架构

- **Summary**: 前端纯 Web，后端是独立的 Rust CLI/daemon 负责队列和同步
- **Effort**: L（5-7 天）
- **Risk**: High（Android 上进程间通信复杂）
- **Pros**: 架构最纯净，管道层完全独立；前端可随时替换
- **Cons**: 初始工作量最大，Android 上跑 Rust daemon 需要额外方案

## Recommended Approach

**方案 A: Tauri 2.0 先跑通 + 冷启动测量**

理由：它是一个"低成本验证实验"。如果 Tauri 在你的 OPPO 上冷启动 < 500ms，你就省了维护两套代码的成本。如果不行，这个实验的代价只是两天时间，而且积累的飞书 API 对接代码在任何方案里都能复用。

## 架构决策（Eng Review 2026-05-16 + Codex Review）

### D1: IPC 模式 → Tauri Command

前端通过 `invoke('send_message', { text })` 调用 Rust `#[tauri::command]` handler。

关键规则：**command 在 SQLite 写入后立即返回**，不等网络同步。返回值包含 message ID 和 `queued` 状态。

Command 返回值语义：
- `queued`：本地写入成功
- `rejected`：空文本或非法输入，不写入

### D2: 同步状态通知 → Command + Event 混合

- Command（invoke）：发送消息，获得 message ID
- Event（emit/listen）：Rust 异步同步完成后通知前端刷新气泡状态

关键规则：**SQLite 是状态真相来源，Event 只是 UI 刷新提示**。Event 丢失不影响数据正确性，前端从 SQLite 重建状态即可。

MVP 阶段 Event 是 per-message 的；批量同步时可改为携带多个 ID 的聚合事件。

### D3: 首屏渲染 → 直接完整加载

个人使用场景消息量小，SQLite 读几百条纯文本 < 10ms。不做异步分片，不做 input-first-then-history 模式。

关键规则：**输入框 focus 不依赖网络操作**。历史加载是纯本地的，token 刷新和同步任务在 UI 可交互后才启动。

### D4: 项目结构 → Cargo Workspace + 独立 feishu-client crate

```
flash-idea/
├── Cargo.toml              # workspace root
├── crates/
│   └── feishu-client/      # 独立 crate，无 Tauri 依赖
│       ├── Cargo.toml
│       └── src/lib.rs      # token 管理 + 文档追加
├── src-tauri/              # Tauri app，依赖 feishu-client
│   ├── Cargo.toml
│   └── src/
└── src/                    # Web 前端
```

独立 crate 的价值（Codex 修正）：**可测试性、解耦、mock 友好**，而非"切 Kotlin 时便宜复用"（UniFFI/JNI 集成成本不低）。

边界划分：
- `feishu-client`：飞书认证 + 文档追加操作
- app 层（src-tauri）：消息队列、重试策略、文档分割、前台/恢复逻辑
- 存储层：SQLite 消息状态

### D5: 冷启动判定标准

- 目标：T4 - T0 < 500ms
- 可接受 MVP 上限：< 800ms（继续优化）
- 迁移触发：持续 > 800ms → 切 Kotlin

### 补充决策（Codex 指出的缺失项）

**凭证存放**：飞书 app_id / app_secret 放本地 `.env` 文件，不编译进 APK。开源后由用户填入自己的凭证，README 引导配置。

**幂等性**：可接受重复段落。崩溃后重试最多多写一行到飞书，个人使用可容忍。

**消息校验**：空文本不发送，不设长度上限，多行输入算一条消息一个段落。

**重试时间**：MVP 用 `retry_count` + App 恢复前台时统一重试。不加 `next_retry_at` 字段。

**观测性**：Day 2 实现 T0-T4 各阶段 timestamp logging，输出到 Android logcat。

## 技术细节补充

### 飞书 API 对接

目标 API endpoint：`POST /open-apis/docx/v1/documents/{document_id}/blocks/{document_id}/children`
- block_type: 2（文本段落）
- document_id 既是文档 ID 也是根 block ID
- 每次追加在文档末尾新增一个段落
- 限频：单应用每 API 上限 3 次/秒

最小 request body 示例：
```json
{
  "children": [{
    "block_type": 2,
    "text": {
      "elements": [{
        "text_run": {
          "content": "[14:32:07] 明天下午三点开会",
          "text_element_style": {}
        }
      }],
      "style": {}
    }
  }]
}
```

### 飞书文档格式设计

MVP 阶段不做复杂格式，像聊天记录一样直接灌入：

```
[14:32:07] 明天下午三点和林韬开会
[14:35:22] 视频吊坠食物识别的 prompt 要换一种写法
[16:01:55] 周末把 Flash Idea 的 Day 1 跑通
```

每条消息就是一个文本段落，格式为 `[HH:MM:SS] 内容`。

### 文档分割策略

按**每日首条消息**分割，切分点是本地时间 **06:00**：
- 每天 06:00 之后的第一条消息 → 创建新文档写入
- 06:00 之前的消息（凌晨熬夜）→ 归入前一天的文档
- 新文档标题格式：`Flash Idea - YYYY-MM-DD`（以切分日期为准）
- 新文档自动创建为飞书知识库中母文档的子节点

为什么不用"6 小时静默间隔"：用户作息不固定（凌晨 0-4 点不等），白天也可能整天不用。
按固定时间点切分更简单可预测，06:00 是一个安全的"新一天"起点。

**知识库结构**：
```
Flash Idea（母文档，用户手动创建）
├── Flash Idea - 2026-05-19
├── Flash Idea - 2026-05-20
└── Flash Idea - 2026-05-21
```

**实现依赖**：飞书知识库 Wiki API
- `.env` 配置母文档的 `node_token`（从 URL 提取）
- 启动时调 `GET /wiki/v2/spaces/get_node` 拿 `space_id`
- 需要新文档时调 `POST /wiki/v2/spaces/{space_id}/nodes` 创建子节点
- 拿到子节点的 `obj_token` 作为 document_id 写入内容

### SQLite Schema（MVP）

```sql
CREATE TABLE messages (
  id TEXT PRIMARY KEY,          -- UUID
  text TEXT NOT NULL,
  created_at TEXT NOT NULL,     -- ISO 8601
  sync_status TEXT DEFAULT 'queued',  -- queued / synced / failed
  retry_count INTEGER DEFAULT 0,
  target_doc_id TEXT,           -- 目标飞书文档 ID。MVP 阶段填 .env 中的固定值，Sprint 2 文档分割时按规则填入新文档 ID
  metadata TEXT DEFAULT '{}',   -- JSON，预留结构化扩展。MVP 阶段写入 '{}'，后续管道对接时再定义 schema
  synced_at TEXT                -- 同步成功的时间戳
);
```

### 同步策略

- **正常**：发送后立即触发前台同步任务，单条调用
- **离线积压**：App 恢复前台时逐条发送，固定 350ms 间隔（保证不超 3次/秒限频）
- **失败重试**：指数退避（1s → 2s → 4s → 8s），最多重试 5 次后标记 failed
- **用户对 failed 消息**：气泡上显示重试按钮，点击重新入队

### Android 后台限制

Tauri App 被切到后台或被杀后，Rust 后端的 worker 不会继续运行（Android Oreo+ 限制后台执行）。MVP 策略：
- 不依赖后台同步，只在 App 前台时同步
- App 恢复前台时检查 queued 消息并批量同步
- 后续如需后台同步，需借助 Android 原生 WorkManager（通过 Tauri plugin 调用）

### 冷启动测量定义

从 **launcher intent 发出** 到 **输入框获得焦点（可输入状态）** 的总时长。细分：
- T0: launcher intent
- T1: Activity onCreate
- T2: WebView 初始化完成
- T3: 前端 first paint
- T4: input focus（可输入）
- 目标：T4 - T0 < 500ms

## Open Questions

1. **Tauri 2.0 Android 冷启动实际表现**：OPPO Find X7 Ultra 的系统 WebView 是基于哪个 Chrome 版本？WebView 版本碎片化是否影响？
2. **飞书 API token 存储**：tenant_access_token 缓存在内存中，App 被杀后重启时需重新获取（异步，不阻塞 UI 渲染）。首次发送时如果 token 为空或过期才触发刷新，刷新期间消息保持 queued 状态。
3. **SQLite 在 Android Tauri 中的路径和权限**：需要确认 Tauri 2.0 Android 的文件系统沙箱路径，确保 SQLite 数据库正确创建和持久化。

## Success Criteria

1. 从点击图标到输入框可输入，冷启动 < 500ms（实测）
2. 发送一条消息 → 出现在飞书文档末尾，延迟 < 3 秒
3. 离线发送的消息在恢复网络后自动同步，不丢数据
4. 安装包 < 10MB

## Distribution Plan

- **Android**: APK 直接安装（个人使用，不上商店）；后续开源后提供 GitHub Releases
- **macOS**: Tauri 构建 .dmg 或 .app
- **CI/CD**: 暂不需要。个人使用阶段手动构建，开源后考虑 GitHub Actions（注意 tauri-action 目前不完全支持移动端自动构建）
